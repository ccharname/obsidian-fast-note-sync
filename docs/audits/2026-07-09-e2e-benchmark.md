# 同步协议流水线化 E2E 联调报告

日期：2026-07-09
服务端：fast-note-sync-service @ HEAD（sqlite 模式，go build 本地二进制，仓库未改动）
客户端：自研 Node.js WS 协议驱动（协议行为源自服务端 `internal/dto`/`action.go`/`ws_note.go`/`ws_sync_download_cache.go` 直读，未依赖 fastnotesync 仓库 node_modules 里并不存在的 `ws` 包，改用 Node 24 原生 `WebSocket`）

## 测试环境

| 项 | 值 |
|---|---|

| 新协议窗口 | `pipeline-window-up: 8`，`pipeline-window-down: 4`（服务端默认值） |

| 协议开关 | 新协议连接 URL 带 `pv=2&pb=0`（`pb=0`：本次只验证窗口流水线，不涉及 protobuf 提前升级，二者相互独立不影响窗口测试结论）；旧协议连接不带 `pv` |

## 矩阵结果

### A. 上行清单分批上传（NoteSync / NoteSyncBatchAck，止于 NoteSyncEnd）

| 模式 | 批大小 | 总批数 | 客户端窗口 | 耗时 | 
|---|---|---|---|---|
| 新协议（pv=2，客户端窗口=8） | 100 | 20 | 8（≈⌈20/8⌉=3 个窗口周期） | **170ms** |
| 旧协议（stop-and-wait） | 100 | 20 | 1（逐批串行） | **1129ms** |

**加速比：6.64×**（理论值 20×50ms=1000ms vs ⌈20/8⌉×50ms=150ms，实测与理论高度吻合，误差来自处理开销）

> 说明：上行窗口完全由客户端驱动（服务端无窗口开关，参见设计 §3.3），因此新旧对比在同一台服务器上通过客户端发送策略切换实现，隔离出"是否流水线"这一个变量。

### B. 下行全量拉取（清库后新客户端首拉，PageAck 驱动）

| 模式 | 页大小 | 总页数 | 服务端协商窗口 | 耗时 |
|---|---|---|---|---|
| 新协议（pv=2） | 200 | 10 | 4（≈⌈10/4⌉=3 个窗口周期） | **234ms** |
| 旧协议（无 pv） | 200 | 10 | 0（stop-and-wait，服务端强制） | **747ms** |

**加速比：3.19×**（理论值 10×50ms=500ms vs ⌈10/4⌉×50ms=150ms；实测比理论略高，主要是每页 200 条正文数据传输 + 处理的额外开销）

> 说明：下行窗口由服务端按连接的 `ProtoVersion` 决定（`ws_note.go:1112-1118`），旧协议连接即便服务端配置了 `pipeline-window-down=4` 也会被强制为 0——这一点本次也顺带验证了。

### C. 正确性断言

两条独立验证路径，结果一致：

| 验证路径 | 新协议 vault | 旧协议 vault | 结论 |
|---|---|---|---|
| 服务端 sqlite DB 直查（`(pathHash,contentHash)` 集合，绕过下载链路） | 2000/2000，与合成清单集合 **完全相等**（`exact_match=True`） | 2000/2000，**完全相等** | ✅ |
| 下载端接收明细集合 vs 上传合成清单 | missing=0, extra=0, match=true | missing=0, extra=0, match=true | ✅ |

结论：**窗口流水线模式与 stop-and-wait 模式的最终数据集合逐字节一致，无丢失、无重复、无损坏**，符合设计 §8 "集合语义不变" 的核心正确性承诺。

### D. 回滚开关验证 —— **发现问题**

设计文档 §8 把"后台将 `pipeline-window-up/down` 设 0 即可秒级回滚到 stop-and-wait"作为高危风险的首要缓解手段。实测：

| 用例 | 配置值（up, down） | 协商块实际返回（up, down） | 是否符合预期 |
|---|---|---|---|
| 主用例（回滚场景） | (0, 0) | **(8, 4)** | ❌ **不符合**——未生效，仍是默认值 |
| 对照用例（非零覆盖） | (2, 3) | (2, 3) | ✅ 符合——非零值可以正常覆盖默认值 |

**根因定位**：`internal/app/config.go:78`

```go
// 第一次 defaults.Set(c) 在 yaml.Unmarshal 之前，为空结构体填充默认值
if err := defaults.Set(c); err != nil { ... }   // config.go:60
err = yaml.Unmarshal(file, c)                    // config.go:69
// 第二次 defaults.Set(c) 在 unmarshal 之后，注释写的意图是"填充 YAML 中存在但值为空的字段"
if err := defaults.Set(c); err != nil { ... }    // config.go:78
```

`creasty/defaults` 库判断"是否需要填充默认值"的依据是**字段是否为该 Go 类型的零值**，而 `int` 类型无法区分"YAML 里显式写的 `0`"和"YAML 里根本没写这个 key"——两者都是零值。所以第二次 `defaults.Set` 会把运维显式写的 `pipeline-window-up: 0` / `pipeline-window-down: 0` 当作"没写"，重新用 struct tag 的 `default:"8"` / `default:"4"` 覆盖回去。**这不是本次窗口流水线功能引入的新代码问题（config.go 的这个二次填充机制是既有基础设施），但设计里承诺的"配置 0 秒级回滚"这条安全阀目前实际不起作用**，一旦线上真的需要紧急回滚窗口流水线，运维按文档把配置设成 0 是没有效果的，仍会保持窗口开启。

**影响范围细化**（代码复核补充，未做运行时 E2E）：
- 管理后台运行时改 0 **能**生效：`handler_admin_control.go:426-435` 用 `*int` 指针语义（nil 判断而非零值判断），显式传 0 会被赋值；协商块读取走 `PipelineWindows()`（`pkg/app/websocket.go:1224`）拿的是运行时内存值。
- 但后台保存会 `cfg.Save()` 落盘 yaml（写入 `pipeline-window-up: 0`），**下一次进程重启时**配置加载的二次 `defaults.Set` 把这个显式 0 又吞回 8/4——即"后台回滚成功 → 服务重启 → 窗口模式静默恢复"，运维不会收到任何提示。容器化部署（纯配置文件启动）则从一开始就无法用 0 关闭窗口。
- 窗口=0 的**行为退化正确性本身没问题**：场景 B 旧协议连接（无 pv，服务端强制 Window=0）走的就是 pump 的退化分支，逐页 stop-and-wait 行为和数据正确性均已实测通过。坏的只是"用配置文件显式写 0"这一个开关路径。

需要人工判断的点（不属于我能替你决定的范围）：这类"运维显式配置的 0 被静默吞掉"的模式，如果 `internal/config` 里还有其他同样用 `default:"non-zero"` 标注、且业务上 0 是合法值（不是"未设置"）的字段，大概率有同样的坑，建议顺手排查一遍（我没有做全仓扫描，只验证了这两个字段）。

## 发现的问题汇总

| # | 问题 | 严重度 | 状态 |
|---|---|---|---|
| 1 | 配置文件显式写 `pipeline-window-up/down: 0` 不生效（被 `internal/app/config.go:78` 的二次 `defaults.Set` 静默覆盖回默认 8/4）。后台运行时改 0 可生效（指针语义），但落盘后**重启即静默恢复窗口模式**；容器化纯配置部署无法关闭窗口 | **高**（设计文档 §8 标注为高危风险的首要缓解手段半失效） | 待修复 |
| 2 | REST 注册/登录发放的 token 是 `p:rest` 专属 scope，无法直接用于 WS 握手，必须额外调用 `POST /api/token` 铸造 `p:ws` scope 的 token | 低（非本次功能引入，是既有接入设计，但文档未清楚说明，建议补充客户端接入文档） | 建议补文档 |

## 结论

- **性能收益符合设计预期**：上行 6.64×、下行 3.19×，均与设计文档理论测算（⌈批数/窗口⌉个 RTT 周期）高度吻合。
- **正确性完全达标**：两种模式下发集合与服务端持久化数据逐字节一致，DB 直查 + 下载端比对双路径均确认。
- **回滚开关存在实现缺陷，需要在上线前修复**，否则设计文档承诺的"无需回滚版本、秒级配置回滚"实际不可用。
